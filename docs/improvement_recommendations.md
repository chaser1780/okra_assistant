# okra assistant 改进建议文档

- 文档定位：面向开发推进的改进清单
- 适用范围：`F:\okra_assistant` 当前独立工程
- 更新日期：2026-03-11

## 1. 文档目的

本文聚焦当前项目在功能逐步丰富后暴露出的几个关键改进方向，重点覆盖以下五类内容：

1. 值得注意的现状
2. 一个很关键的金融准确性观察
3. 从开发者角度的改进建议
4. 从金融从业者角度的改进建议
5. 从用户角度的改进建议

目标不是否定现有实现，而是把下一阶段最值得投入的事项按工程语言整理清楚，便于后续逐项落地。

---

## 2. 值得注意的现状

### 2.1 项目已经形成了完整闭环，但层次越来越多

当前系统已经不是单一脚本，而是由以下链路组成的本地研究系统：

- 日内分析链
- 实时收益链
- 夜间复盘链
- 手工交易录入与持仓回写链
- 桌面端查看与手动触发链

这说明项目方向是成立的，但也意味着后续维护难点已经从“能不能做出来”转移到“结构能不能长期支撑”。

### 2.2 主链路与保留链路并存，存在“配置漂移”

当前代码中已经出现“主链路正在使用”和“历史保留/备用实现”并存的情况：

- `score_portfolio.py` 仍存在，但当前主日内链路并不依赖它
- `agents.toml` 中有编排配置，但实际并行策略已写入代码，配置未完全接线
- `review.toml` 中定义了多周期复盘，但当前夜间主链路仍主要跑 `horizon=0`
- `realtime_valuation.toml` 存在，但实时收益主链路尚未完整读取这份配置

这类漂移短期问题不大，但长期会让“看配置”和“看代码”得出的结论不一致，增加维护成本。

### 2.3 桌面壳已经可用，但单文件体量偏大

桌面端当前已具备：

- 任务触发
- 状态展示
- 实时收益查看
- 研究输出浏览
- 夜间复盘查看
- 交易录入与持仓回写

但 `desktop_shell.py` 已承担：

- UI 布局
- 任务执行
- 运行状态管理
- 页面刷新逻辑
- 数据展示格式化

这在现阶段提高了开发速度，但继续扩展会让单文件维护难度明显上升。

### 2.4 当前外部数据源较多，但适配层仍偏脚本化

项目当前已经接入了多种外部来源：

- Eastmoney 历史净值
- Eastmoney 基金公告/文章
- Sina 盘中代理行情
- `fundgz.1234567.com.cn` 实时估值
- Tabcode 中转的大模型接口

这是系统能力的重要来源，但适配方式仍主要分散在各个脚本内部。后续若增加替代源、A/B 源或降级策略，当前结构会逐步变难维护。

### 2.5 当前系统已具备追溯性，但缺少测试化保障

优点：

- 多数步骤都有 JSON 落盘
- 多智能体输出可回放
- 最终建议和复盘报告可追溯
- 快照目录可做版本比较

不足：

- 缺少自动化测试
- 缺少契约校验测试
- 缺少主链路回归测试
- 缺少针对模型输出结构的稳定性测试

当前更像“高质量脚本工程”，还未完全进入“可持续演进的工程产品”阶段。

### 2.6 文档与工程卫生还有遗留项

当前还存在一些明显可收口的问题：

- 个别文档仍有乱码遗留
- 旧配置与新实现之间未完全统一
- 启动脚本中 Python 路径仍为硬编码
- 部分运行参数既写在配置里又写在代码里

这些问题不影响主功能，但会影响交接、迁移和后续继续开发。

## 3. 一个很关键的金融准确性观察

### 3.1 当前“组合持仓市值”没有形成稳定的每日官方净值重估闭环

这是当前最值得优先关注的金融准确性问题。

目前系统已经具备：

- 盘中实时收益估算
- 交易后持仓即时回写
- 夜间复盘按真实涨跌判断建议效果

但 `portfolio.json` 中的这些核心字段：

- `current_value`
- `holding_pnl`
- `holding_return_pct`
- `total_value`

并没有形成“每日收盘后按官方净值自动重估”的稳定闭环。

当前更多依赖的是：

- 初始人工录入
- 交易录入后的回写
- 部分脚本对份额和净值的同步

这意味着：

1. 实时收益链路本身可以比较准确地估算“今天盘中变化”
2. 但组合层长期市值、仓位占用、上限判断、收益率基线，可能会逐步偏离真实情况

### 3.2 这个问题的影响比表面看起来更大

它不仅影响“显示对不对”，还会影响：

- 单基金是否接近上限
- 资金仓可调拨金额判断
- 增减仓建议的边际空间
- 复盘时对仓位变化背景的理解
- 组合总收益、总资产和风险暴露判断

换句话说，如果没有“每日官方净值重估”，系统会逐渐出现一种现象：

- 盘中层面看起来很准
- 组合层面却慢慢不准

这会削弱整个系统后续做资产配置、组合风险控制和中周期复盘的可信度。

### 3.3 建议将其定义为高优先级改造项

建议新增一个明确的“收盘后组合重估步骤”，在夜间链路或独立维护链路中执行：

- 读取当天官方净值
- 按持有份额重算每只基金 `current_value`
- 更新 `holding_pnl / holding_return_pct`
- 汇总更新组合 `total_value / holding_pnl / as_of_date`
- 将这一步落盘并形成日志

如果这一项补上，整个项目的金融可信度会明显上一个台阶。

## 4. 从开发者角度的改进建议

### 4.1 用显式数据模型替代大量裸字典

当前系统的很多核心对象都以裸 `dict` 流转：

- `llm_context`
- agent output
- final advice
- validated advice
- realtime item
- review memory

建议逐步引入：

- `dataclass`
- 或 `pydantic`

优点：

- 字段边界更清晰
- IDE 支持更好
- schema 漂移更容易发现
- 单元测试更容易写

### 4.2 收敛配置漂移

建议对以下配置做一次专项整理：

- `agents.toml`
- `review.toml`
- `realtime_valuation.toml`

原则只有两个：

1. 已经不用的配置就删掉
2. 还要保留的配置必须真正接入代码

不要长期维持“配置看起来可调，但实际上代码没在用”的状态。

### 4.3 增加主链路自动化测试

建议最少补三类测试：

- **契约测试**：验证 JSON 输出结构是否符合预期
- **快照测试**：固定 mock 输入，对比 `validated_advice`、`review_memory`、`realtime_monitor`
- **编排测试**：验证主链路关键步骤是否按预期执行

优先顺序建议：

1. `validate_llm_advice`
2. `build_realtime_profit`
3. `update_portfolio_from_trade`
4. `review_advice`
5. `run_multiagent_research` 的 mock 路径

### 4.4 拆分桌面壳

建议逐步把桌面壳拆成几层：

- `app/services`：运行任务、读取文件、解析状态
- `app/state`：页面状态和任务状态
- `app/views`：各页面布局
- `app/formatters`：金额、日期、字段展示格式化

这样做的意义不是“为了优雅”，而是为了保证之后继续加：

- 图表
- 多日期对比
- 筛选
- 任务历史
- 更多页面

时不会把 `desktop_shell.py` 继续堆成更大的单文件。

### 4.5 统一外部数据源适配层

当前建议把行情与资讯抓取逐步抽成 provider adapter 模式：

- Quotes provider
- News provider
- Proxy provider
- Estimate provider

每个 provider 至少统一：

- 输入参数
- 输出字段
- 失败兜底
- stale 判定
- provider name

这样未来切换来源时，不需要重写业务层。

### 4.6 增加运行元数据落盘

目前已经有日志和 timing 输出，建议再新增结构化运行元数据，例如：

- 每次任务唯一 run id
- 开始/结束时间
- 每步耗时
- 失败步骤
- 使用模型
- 使用 transport
- agent 时延分布

这会非常利于后续做：

- 性能分析
- 故障定位
- 稳定性回归
- UI 状态展示

### 4.7 清理文档与工程卫生

建议单独做一轮工程卫生收口：

- 修复乱码文档
- 删除未使用代码或明确标注“保留模块”
- 统一脚本中的 Python 路径策略
- 给 docs 建立索引文档

这会显著改善项目的“可交接性”。

## 5. 从金融从业者角度的改进建议

### 5.1 优先补“每日官方净值重估”

这是当前金融准确性提升的第一优先级。

建议加入夜间固定步骤：

- 以持有份额为基础
- 使用官方净值
- 重算组合与单基金持仓市值

这是整个系统从“研究工作台”向“可信组合助手”升级的关键。

### 5.2 实现真正的多周期复盘

当前复盘更多偏日内/次日层面。

建议把复盘扩展到：

- `T+1`
- `T+5`
- `T+20`

原因：

- 主题基金和主动权益很多时候并不适合只用次日涨跌评价
- 单日结果很容易把“时点问题”和“研究方向错误”混为一谈

多周期复盘后，系统对“信号周期”的理解会更专业。

### 5.3 增强组合暴露分析

目前已有 `role` 和 `style_group`，但仍不够。

建议增加：

- 主题集中度
- 风格集中度
- 海外/国内市场暴露
- QDII 汇率/时差风险
- 单一高波动主题叠加程度

这样 `risk_manager` 的判断会更像真正的组合风控，而不只是“更保守一点的研究经理”。

### 5.4 引入基金慢变量数据

当前 prompt 已经在尝试让 agent 讨论慢变量，但真实数据层仍偏弱。

建议后续补充：

- 基金经理任职稳定性
- 规模变化
- 回撤特征
- 费率
- 风格漂移
- 同类相对表现

尤其是主动基金，如果没有慢变量，很多“中期判断”只能停留在经验性描述。

### 5.5 更真实地建模交易约束

后续建议在策略层显式加入：

- 持有期限制
- 赎回到账时间
- 场外基金确认时滞
- QDII 净值滞后与时差
- 转换规则与手续费影响

这样系统给出的建议会更接近真实执行环境。

### 5.6 增加“无操作基线”对照复盘

建议复盘时不仅比较：

- “做了这笔建议后结果如何”

还比较：

- “如果当天什么都不做，结果如何”

这能更准确评价系统到底是在创造价值，还是只是制造交易。

## 6. 从用户角度的改进建议

### 6.1 首页优先回答“今天最重要的三件事”

建议把 Dashboard 再进一步产品化：

1. 今天最需要关注什么
2. 今天建议有没有变
3. 今天到底要不要操作

这比把大量细节同时铺出来更符合实际使用习惯。

### 6.2 增加“建议变化解释”

用户最常见的问题不是“今天建议是什么”，而是：

- 为什么和昨天不一样
- 为什么昨天看多今天又观望
- 为什么只减不加

建议增加：

- 相比上一日建议的变化说明
- 触发变化的核心证据
- 被哪些约束压制/修改

### 6.3 更清楚地区分三类时间

建议在 UI 中始终显式区分：

- 当前查看日期
- 任务运行日期
- 数据实际对应日期/行情时间

特别是实时页，需要用户一眼看懂：

- 官方净值是哪一天
- 实时估值是什么时间
- 代理行情是什么时间

### 6.4 增加交易后的前后对比

当前交易录入已经能回写持仓，但对普通用户来说还不够直观。

建议增加录入后对比：

- 交易前市值
- 交易后市值
- 资金仓变化
- 份额变化
- 单只基金距离上限还有多少

这样用户更容易建立对系统的信任。

### 6.5 增加异常提醒而不只是静态报告

建议加轻量级提示，例如：

- 今日报告尚未生成
- 某个关键 agent 失败
- 实时估值 stale
- 夜间复盘尚未完成
- 资金仓低于目标底仓

用户不一定会主动翻到日志，但会非常需要“哪里不正常”的明确提示。

### 6.6 增加通俗版解释模式

当前系统已经非常强，但对非技术用户或非专业用户来说有一定门槛。

建议增加“简版解释”：

- 结论
- 原因
- 风险
- 建议是否需要今天执行

把专业结构保留给高级页面，把日常使用体验做得更直观。

## 7. 建议优先级

### P0（建议优先尽快落地）

1. 每日官方净值重估组合
2. 配置与代码对齐，清理漂移配置
3. 主链路 JSON 契约测试
4. 时间语义和 stale 提示统一

### P1（建议作为下一阶段核心工作）

1. 多周期复盘
2. 桌面壳拆层
3. 运行元数据结构化
4. 外部数据源适配层统一
5. 组合暴露分析增强

### P2（建议作为产品化升级）

1. 建议变化解释
2. 交易前后对比
3. 通俗版解释模式
4. 更多图表与运行历史视图

---

## 8. 建议的推进顺序

为了兼顾金融可信度、工程可维护性和用户体验，建议按以下顺序推进：

### 第一阶段：先补准确性

- 每日官方净值重估
- stale 与时间语义统一
- 多周期复盘基础框架

### 第二阶段：再补工程稳态

- 配置漂移清理
- 测试体系建设
- 运行元数据结构化
- 桌面壳拆层

### 第三阶段：最后补产品体验

- 建议变化解释
- 更好的 Dashboard
- 交易前后对比
- 通俗模式

---

## 9. 总结

`okra assistant` 当前已经具备很强的雏形：

- 研究链路完整
- 委员会机制清晰
- 复盘记忆闭环已建立
- 本地桌面工作台已经可用

当前最需要的不是继续无节制扩功能，而是把“金融准确性、配置一致性、测试保障、用户语义清晰度”四件事补扎实。

如果这四件事做好，项目会从“功能很多的本地研究系统”，进一步升级成“可信、可持续演进、可长期使用的个人基金研究平台”。
