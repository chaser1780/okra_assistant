# okra assistant 下一阶段落地方案

- 日期：2026-03-19
- 目的：基于当前 `F:\okra_assistant` 的真实代码、真实数据产物和外部产品调研，给下一幕的具体改动提供可直接执行的路线图
- 适用对象：
  - 个人基金投资者
  - 同时希望保留研究/调试能力的进阶用户

## 1. 先给结论

当前项目已经不是“基金信息面板”，而是一个具备以下闭环能力的本地基金助手雏形：

- 日内抓取与实时估值
- 多智能体研究与委员会式建议
- 规则校验与交易闭环
- 夜间复盘与记忆更新
- 本地桌面工作台

但它还没有完全变成“个人基金管理助手”，核心原因不是功能少，而是功能层次还没有完全收束。

当前最值得优先解决的 6 件事：

1. **统一建议口径**：`score_funds.py` 的日报打分结论，与 `validated_advice` 的最终可执行建议存在冲突。
2. **补齐长短期策略分层**：当前系统擅长“今天怎么动”，但还不擅长“长期怎么配、何时再平衡、什么属于不该频繁动的仓位”。
3. **把组合管理放到基金管理前面**：当前组合有 11 只 `tactical` 基金、战术仓约占 49.10%，高波动主题占比 52.0%，但系统前台仍以逐基金建议为主，组合层目标不够显式。
4. **压缩多智能体噪音**：`2026-03-19` 的 `aggregate.json` 已出现明显冗长、重复和尾部填充内容，拖累展示、成本和可读性。
5. **把界面从“分析台”收束成“助手”**：现在的 8 个标签页更像研究工作台，不像个人投资助手；应依赖已有的 `view_mode` 做双层界面。
6. **把导入和维护成本降下来**：当前持仓维护仍偏手动，适合熟悉项目的人，不够适合日常基金用户。

## 2. 基于当前项目的真实判断

### 2.1 当前项目的优势

从代码和数据产物看，项目的强项很明确：

- `scripts/run_daily_pipeline.py` 已经把日内、夜间、复盘编排成完整链路。
- `scripts/validate_llm_advice.py` 已经具备“LLM 建议 -> 规则校验 -> 可执行动作”的硬约束层。
- `reports/daily/2026-03-19_portfolio.md` 已经能输出组合暴露、风险预算、执行清单。
- `app/desktop_shell.py` 与 `app/views/` 已具备桌面端工作台形态。
- `db/review_memory/memory.json` 已形成持续积累的复盘记忆，而不是一次性分析。

这说明下一阶段不应再从“多加几个功能”入手，而应从“重新组织现有能力”入手。

### 2.2 当前最关键的业务问题

#### 问题 A：前台存在双建议源，用户会看到互相打架的结论

当前至少有两套前台结论来源：

- `scripts/score_funds.py`
  - 基于日涨跌、月涨跌、新闻条数做阈值打分，输出 `add / observe / reduce / avoid`
- `db/validated_advice/2026-03-19.json`
  - 基于委员会输出和规则校验，最终只给出 2 笔减仓和 2 笔固定定投

`reports/daily/2026-03-19.md` 仍显示：

- `add`：2 只
- `reduce`：8 只

但 `db/validated_advice/2026-03-19.json` 的最终可执行动作是：

- `017193` 减 300
- `020274` 减 100
- `017641` 定投 25
- `016452` 定投 25

这会让用户在“日报”“研究建议”“交易执行”之间感受到口径冲突。对个人助手来说，这是最高优先级问题。

#### 问题 B：系统有“短线动作能力”，但没有真正的“长期资金管理能力”

当前策略层主要是：

- `core_dca`
- `tactical`
- `cash_hub`
- `fixed_hold`

这能表达基础角色，但还不够表达：

- 长期核心配置目标
- 中期卫星仓目标
- 再平衡触发阈值
- 短线动作预算上限
- 哪些建议只是观察，不应频繁交易

结果是系统更像“盘中研究器”，还不是“长期可持续使用的基金助手”。

#### 问题 C：当前组合结构过于分散，系统前台却没有主动帮用户做收敛

基于当前 `portfolio.json` 和组合报告：

- `tactical` 基金：11 只
- 战术仓占比：49.10%
- 高波动主题占比：52.0%
- 海外权益占比：20.8%
- 多只战术仓当前市值低于 600 元

这意味着当前用户更需要的是：

- 哪些仓位该合并
- 哪些主题重叠过高
- 哪些仓位应该进入长期持有层
- 哪些仓位应该明确成为短期观察仓

而不是继续给 11 只战术基金分别讲很长的故事。

#### 问题 D：多智能体产出已经开始影响系统可用性

当前 `2026-03-19` 的数据规模：

- `db/agent_outputs/2026-03-19/aggregate.json` 约 120.9 KB
- `db/llm_raw/2026-03-19.json` 约 1403 KB

同时 `aggregate.json` 中已经出现：

- 长段重复总结
- “完成”“结束”“以下补充”等尾部口水话
- 连续空白字符串填充

这类问题如果不处理，会直接带来：

- token 成本升高
- 建议生成更慢
- UI 默认详情过长
- 复盘噪音增多

#### 问题 E：界面结构更像“全量后台”，不是“个人用户主路径”

当前标签页：

- 总览
- 运行
- 实时监控
- 研究建议
- 智能体
- 复盘记忆
- 交易执行
- 设置数据

这套结构适合开发/分析，但不适合普通个人基金用户每天打开就用。

而且当前代码层面虽然已经拆了 `views/`，但核心 UI 仍较重：

- `app/desktop_shell.py`：1477 行
- `app/ui_support.py`：784 行
- `app/ui_schemas.py`：416 行
- `app/ui_schema_overrides.py`：485 行

尤其 `ui_schemas.py` 与 `ui_schema_overrides.py` 存在同名 schema builder 并行维护，说明展示层还没有真正收口。

## 3. 外部参考里最值得借鉴的部分

- 强调 reporting period、performance、drawdown、risk indicators

- 增加 30/90/180 天收益、回撤、波动、相对基准

- 明确 okra 的产品定位是“本地个人基金管理助手”

#### [x2rr/funds](https://github.com/x2rr/funds)

- 增加更轻的“持仓概览 + 实时涨跌 + 快速导入”
- 增加截图/OCR 或 CSV 导入，降低初次接入门槛

## 4. 下一阶段产品定位

### 4.1 建议的产品定义

把 okra 定义为：

**“本地运行的个人基金管理与建议助手，优先解决长期配置、短期动作、风险暴露和执行闭环。”**

它不应该继续主要被定义为：

- 多智能体实验台
- 纯日报生成器
- 只面向分析过程的后台

### 4.2 两种使用模式

继续保留现在已有的 `view_mode`，但把它从“文案视角切换”升级为“产品层级切换”：

- 投资用户模式
  - 默认只看 4 个主页面
  - 聚焦今日动作、长期配置、风险提醒、复盘结果
- 分析师模式
  - 展开研究建议、智能体、运行与设置
  - 允许查看原始文本和调试信息

## 5. 长期 / 短期策略框架建议

### 5.1 新的策略分层

建议将现有角色扩成两个维度：

- 资金层：
  - `core_long_term`
  - `satellite_mid_term`
  - `tactical_short_term`
  - `cash_defense`
- 时间层：
  - `long_term`
  - `swing`
  - `intraday_observation`

现有角色可先这样映射：

- `core_dca` -> `core_long_term`
- `fixed_hold` -> `cash_defense` 或 `core_defense`
- `cash_hub` -> `cash_defense`
- `tactical` -> 先拆成 `satellite_mid_term` 与 `tactical_short_term`

### 5.2 面向 balanced 风险画像的默认配比建议

以下建议应做成“默认模板，可调整”，不是硬编码结论：

- 长期核心仓：45% - 60%
  - 宽基、长期指数、核心全球指数类
- 防守/现金仓：20% - 30%
  - 现金管理、短债、稳健债基
- 卫星主题仓：15% - 25%
  - AI、制造、商品、行业主题
- 短期战术仓：5% - 15%
  - 只保留高置信、可快速验证的主题动作

对照当前组合，系统应该默认给出结构提醒：

- 当前战术主题数量偏多
- 高波动主题占比较高
- 更适合先做“合并与收敛”，再做更细动作

### 5.3 长期策略建议能力

长期策略模块要解决的不是“今天买不买”，而是：

- 目标资产配置是什么
- 当前偏离了多少
- 本月是否需要再平衡
- 哪些基金属于长期定投，不应被短期噪音改写

建议系统默认输出：

- 目标配比
- 当前配比
- 偏离程度
- 再平衡建议
- 定投计划是否维持

### 5.4 短期策略建议能力

短期建议不要再平均覆盖所有基金，只输出高质量动作。

建议强制加入以下门槛：

1. 数据新鲜度达标
2. 委员会共识强
3. 与长期配置不冲突
4. 组合拥挤度允许
5. 行动金额和交易次数在预算内

默认只给：

- 0 - 3 个高优先动作
- 每个动作明确：
  - 时间窗口
  - 资金来源
  - 为什么现在动
  - 什么情况下取消

### 5.5 对当前组合最适合的策略产品化方向

不是继续把 11 只战术仓都做成等权“建议对象”，而是让系统主动帮助用户完成：

- 主题收敛
- 角色重分类
- 核心仓与战术仓边界划清
- 卖出资金优先回现金还是转长期仓

## 6. 具体功能改造方案

### 6.1 方案一：统一建议引擎

目标：

- 彻底消除“日报”和“最终建议”口径冲突

做法：

1. `score_funds.py` 不再输出前台动作结论
2. `score_funds.py` 只保留为低层 signal 特征生成器
3. `build_daily_report.py` 改为直接消费 `validated_advice`
4. 日报里如果要保留“原始评分”，只能作为次级字段，不得再用 `add / reduce / avoid`

受影响文件：

- `scripts/score_funds.py`
- `scripts/build_daily_report.py`
- `scripts/build_llm_context.py`
- `reports/daily/*.md`

验收标准：

- Dashboard、日报、研究建议、交易页的动作数量完全一致
- 用户再也不会看到“日报建议买、最终建议却 hold”的冲突

### 6.2 方案二：引入组合目标与再平衡引擎

目标：

- 让系统真正支持长期基金管理

做法：

1. 在 `strategy.toml` 中新增：
   - 长期核心仓目标比例
   - 防守仓目标比例
   - 卫星仓上限
   - 短期战术仓上限
   - 再平衡阈值
2. 新增组合层决策：
   - 当前偏离
   - 是否需要再平衡
   - 哪个 bucket 资金过高/过低
3. 将“按基金建议”升级为“先按 bucket 建议，再到基金”

建议新增或改造文件：

- `config/strategy.toml`
- `scripts/portfolio_exposure.py`
- `scripts/validate_llm_advice.py`
- `scripts/build_portfolio_report.py`

验收标准：

- 组合层先有“结构建议”，再有“基金动作”
- 长期仓不会被短线动作频繁打断

### 6.3 方案三：把委员会输出压缩成“动作卡片”

目标：

- 让 AI 能力服务执行，而不是制造阅读负担

做法：

1. 委员会与 manager 输出增加更强长度约束
2. 新增输出后处理：
   - 去尾部口水话
   - 去空字符串
   - 截断重复句
3. 对 UI 默认只暴露：
   - 一句话结论
   - 3 条依据
   - 2 条风险
   - 1 个执行提示
4. 原始文本下沉到“调试/原文”折叠区

受影响文件：

- `scripts/run_multiagent_research.py`
- `scripts/generate_llm_advice.py`
- `app/ui_schema_overrides.py`
- `app/widgets.py`

验收标准：

- 研究页首屏不再出现超长文本墙
- 聚合输出不再出现“完成”“结束”“空白填充”

### 6.4 方案四：新增真正的 Portfolio 页面

目标：

- 把组合管理前置，而不是藏在 Dashboard 右侧

建议页面内容：

- 目标配置 vs 当前配置
- 核心 / 卫星 / 战术 / 防守四层分布
- 集中度与重叠提醒
- 再平衡建议
- 本月定投执行情况
- 资金可用性与未来到账节奏

受影响文件：

- `app/desktop_shell.py`
- 新增 `app/views/portfolio.py`
- `app/ui_schema_overrides.py`

验收标准：

- 用户不切换研究页，也能知道“我现在整体仓位是不是合理”

### 6.5 方案五：把主界面压缩成 4 个用户主页面

目标：

- 让桌面端更简洁、更实用

投资用户模式建议保留：

1. 首页
2. 组合
3. 今日动作
4. 复盘

分析师模式再展开：

- 实时监控
- 智能体
- 运行
- 设置数据

这样既保留专业能力，又不会让普通用户每天在 8 个标签页之间来回切。

### 6.6 方案六：补一个低门槛导入入口

目标：

- 把“开始用”这件事做简单

建议支持三种导入：

1. CSV 导入
2. 手工粘贴持仓
3. 截图/OCR 导入

建议新增：

- `scripts/import_portfolio_snapshot.py`
- 设置页中的“导入持仓”入口

验收标准：

- 新用户 3 分钟内可完成初始持仓导入

## 7. 界面简化方案

### 7.1 首页

首屏只回答四件事：

- 今天是否要操作
- 最值得做的 1-3 个动作
- 当前最大风险
- 建议可信度是否足够

建议布局：

- 顶部 Hero：今日结论 + 最后更新时间 + 数据可信度 + 两个主按钮
- 中部左：今日动作卡
- 中部右：风险与暴露卡
- 底部：异常提醒与复盘摘要

不要在首屏默认展示：

- 原始长文本
- 所有 hold 标的
- 智能体级细节

### 7.2 组合页

建议做成真正的“基金管理”中心，而不是被 Settings 吃掉。

要展示：

- 资金层分布
- 目标偏离
- 高重叠主题
- 当前可腾挪资金
- 再平衡建议
- 长期仓与短期仓边界

### 7.3 今日动作页

默认只展示：

- 可执行动作
- 原因
- 风险
- 执行状态
- 交易后影响

将 `hold` 列表下沉到折叠区：

- “今天暂不动作的基金”

### 7.4 复盘页

默认展示：

- 30 / 90 天命中率
- no-trade baseline
- 主要 lessons
- 最近误判类型
- 哪类动作有效、哪类动作应减少

### 7.5 视觉规范

建议不是重做成更花哨，而是更克制：

- 默认少卡片、少长文、少重复标题
- 使用状态色表达“动作 / 风险 / 时效 / 可信度”
- 原始文本默认折叠
- 按“结论 -> 理由 -> 风险 -> 原文”顺序展开

## 8. 下一幕最值得直接开做的改动顺序

## 10. 完成定义

本轮方案落地完成，至少应满足以下标准：

1. 首页、日报、研究建议、交易页对“今日动作”的数量和方向完全一致。
2. 系统能同时输出：
   - 长期配置建议
   - 短期动作建议
3. 投资用户模式下的主导航不超过 4 个页面。
4. 组合层可直接看到目标偏离和再平衡建议。
5. 多智能体默认输出不再出现长尾重复文本和空白填充。
6. 新用户可以通过导入功能快速建立初始组合。
