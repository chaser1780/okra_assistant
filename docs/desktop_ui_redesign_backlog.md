## 1. 文档目标

这份 backlog 只解决一件事：把当前桌面工作台的界面做得更清楚、更专业、更好用，并且让后续界面迭代不再被单文件结构卡住。

它不是泛泛的 UI 建议，而是可直接排期和实施的任务清单。每个任务都包含：

- 明确的用户问题
- 影响页面
- 可执行的改造步骤
- 清晰的验收标准

## 2. 范围边界

### 本次纳入

- 页面信息架构
- 导航与分区
- 卡片、列表、表格、表单、状态反馈的界面形态
- 搜索、筛选、排序、空状态、加载态、错误态

### 本次不纳入

- 估值、复盘、交易回写算法

### 执行约束

- 默认继续基于 Tkinter/ttk 做界面改版，不做整站框架重写。
- 只在“为了支撑界面改造而必须拆分代码”的前提下调整工程结构。
- 所有任务都必须保持现有数据输入输出兼容，不因为改界面而改业务语义。

## 3. 当前界面基线

基于 `app/desktop_shell.py` 和 `app/ui_support.py` 的现状，当前桌面端已经具备完整工作台形态，但存在以下共性问题：

1. 页面很多，但首屏信息优先级不够明确。
2. 多个页面采用“左列表 + 右长文本”的同构布局，阅读密度高但扫读效率低。
3. 详情主要依赖 `build_*_text()` 拼接长文本，导致重点信息缺少视觉锚点。
4. 列表区缺少搜索、筛选、排序和批量状态识别，定位效率低。
5. 成功/失败反馈高度依赖 `messagebox`，连续操作时容易打断节奏。
6. 标签命名、字体层级、颜色语义、状态表达仍不统一。
7. `desktop_shell.py` 同时承载布局、交互、任务状态和页面刷新逻辑，界面继续演进的边际成本较高。

## 4. 改版目标

本轮界面改版要达到以下结果：

1. 用户打开应用后 5 秒内能知道“今天要不要动、为什么动、风险在哪、数据靠不靠谱”。
2. 用户在 Research、Agents、Realtime 三页内，最多 3 次交互就能定位到目标项。
3. 交易前需要确认的风险、约束和仓位变化，在首屏即可读，不依赖通读说明文本。
4. 金融专业用户可以用界面直接完成“建议-执行-风险-结果”的联查。
5. 视觉层级清晰，不再是同质化的信息盒子堆叠。
6. UI 文件结构支持后续继续添加图表、过滤器和新页面，而不是继续堆进一个文件。

## 6. 用户视角 Backlog

### U1. Dashboard 首屏改成“今日决策面板”

- 状态：`[todo]`
- 优先级：`P0`
- 预计工作量：2 天
- 影响页面：Dashboard、顶部 Hero、底部状态栏
- 用户问题：打开应用后要在 Hero、指标卡、长摘要、底部状态栏之间来回扫，首屏没有形成完整闭环。
- 交付物：
  - 首屏决策面板
  - 固定的“今日是否操作”区
  - 固定的“风险与可信度”区
  - 两个主操作按钮
- 执行步骤：
  1. 合并顶部 Hero、Dashboard 指标卡、底部状态信息为一个首屏决策区域。
  2. 首屏只保留四个一级问题：“今天要做什么”“今天可以等什么”“今天不建议做什么”“今天的建议靠不靠谱”。
  3. 将 `self.dash_text` 的长文本下沉到二级详情区，不再占据首屏中心位置。
  4. 把“回到今天”“刷新”“运行日内链路”“进入交易台”放到首屏行动区，避免用户先找按钮。
  5. 增加“最后更新时间”“当前查看日期”“组合估值日期”三类时间标签，并保持并列展示。
- 验收标准：
  - 用户进入 Dashboard 后无需切换标签页，就能判断今天是否需要执行操作。
  - 首屏不出现超过两屏高度的连续长文本。
  - 关键按钮都位于首屏可见区域。

### U4. 成功与失败提示改成页内通知流

- 状态：`[todo]`
- 优先级：`P1`
- 预计工作量：1 天
- 影响页面：运行状态、Trade Console、Settings & Data、全局交互
- 用户问题：`messagebox` 会阻断流程，连续保存或连续运行时体验割裂。
- 交付物：
  - toast 通知组件
  - 页面内成功/失败消息区域
  - 可点击查看日志的失败提示
- 执行步骤：
  1. 为“保存成功”“任务完成”“任务失败”“输入校验失败”建立统一通知机制。

- 验收标准：
  - 常规成功操作不再弹出模态框。
  - 用户连续录入交易、保存设置、刷新数据时不会被多次打断。
  - 失败提示在 1 次点击内可追到日志或出错位置。
